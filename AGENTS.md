# AGENTS.md — автономная разработка релизной версии платформы «Россия 2050»

Этот файл является главным набором инструкций для Codex App / Codex CLI / иных кодовых агентов, работающих с репозиторием платформы. Перед началом любой задачи агент обязан прочитать этот файл, затем файлы из `.codex/tasks/`, `.codex/subagents/`, `docs/PROJECT_CONTEXT.md`, `docs/ACCEPTANCE_CRITERIA.md` и `docs/RELEASE_PROTOCOL.md`.

## 0. Роль агента

Ты работаешь не как помощник, который делает одну мелкую правку, а как автономная команда разработки, ревью и релизной полировки. Твоя задача — довести проект до самодостаточной релизной GitHub Pages-платформы, которую можно открыть локально и показать руководителю без внешнего интернета.

Работай долго, последовательно, с самопроверкой. Не останавливайся после первого успешного запуска. После каждой крупной правки запускай проверки, читай ошибки, исправляй, повторяй цикл до чистого результата.

## 1. Главная цель проекта

Собрать самодостаточную статическую платформу:

**«Россия 2050: расчётная демографическая платформа»**

Шесть страниц:

1. `СКР`
2. `Расселение`
3. `Усадьба`
4. `Маткапитал`
5. `Ипотека`
6. `Выплаты`

Платформа должна отвечать на управленческий вопрос: какие меры, связанные с рождаемостью, жилищными условиями, ИЖС, расселением, ипотекой, выплатами и семейным пространством, позволяют приблизиться к рамке «Россия 2050».

## 2. Неприкосновенные продуктовые требования

1. Горизонт всех расчётов и графиков — только до 2050 года включительно.
2. Никакого 2100 года в пользовательском интерфейсе и runtime-данных.
3. Никаких внешних runtime-зависимостей: ни CDN, ни GitHub Raw, ни API, ни удалённых шрифтов, ни удалённых картинок.
4. Все библиотеки, данные и ассеты должны лежать внутри `docs/`.
5. Платформа должна работать из локального сервера:
   `python -m http.server 8000`
6. Все видимые тексты интерфейса — на русском языке.
7. Никаких служебных слов в интерфейсе: `debug`, `TODO`, `prototype`, `baseline`, `RPN`, `download`, `CSV`, `XLSX`, `fetch`, `GitHub Raw`, `NaN`, `undefined`, `null`.
8. Целевые значения Царьграда — федеральная рамка России, а не обязательная индивидуальная норма для каждого субъекта.
9. Страница СКР должна разделять режимы: Россия / федеральный округ / субъект РФ.
10. РПН-2022 показывать как федеральный слой. Региональные данные РПН использовать только для 2012/2017, если они уже корректно сопоставлены.

## 4. Разрешённые технологии

Проект является статическим сайтом. Разрешены:

- HTML, CSS, JavaScript без серверной части;
- локальный Plotly для графиков;
- локальный Three.js для трёхмерной усадьбы;
- локальные JSON/CSV для данных;
- Python-скрипты для сборки, проверки и подготовки данных;
- Node/Playwright только для разработки и тестов, не для runtime.

## 5. Runtime-запреты

Запрещены в runtime-коде:

- `https://raw.githubusercontent.com/...`
- `https://cdn.jsdelivr.net/...`
- `https://unpkg.com/...`
- `https://cdnjs.cloudflare.com/...`
- `https://cdn.plot.ly/...`
- внешние шрифты Google Fonts;
- внешние изображения;
- внешние API.

Если агенту нужно взять данные из внешнего источника во время разработки, он может сделать это в ETL-скрипте, сохранить локальный итоговый JSON/CSV и убрать runtime-запрос.

## 7. Данные и методология

### 7.1. Авторский прогноз

Нельзя подтягивать данные из внешнего репозитория пользователя во время работы платформы. Нужно создать локальные файлы:

```text
docs/data/author_tfr_forecast_2050.json
docs/data/author_population_forecast_2050.json
docs/data/forecast_join_audit.json
docs/data/forecast_join_audit.csv
```

Источник для разработки: репозиторий пользователя `Population-Forecast-Dashboard-of-Russia-2100`. Runtime-запросов к нему быть не должно.

### 7.2. РПН

РПН-2022 — федеральная оценка желаемого/ожидаемого числа детей и жилищных барьеров. Не строить карту субъектов по РПН-2022 без безопасного ключа субъектов.

РПН-2012/2017 — допустимая региональная ретроспектива, если данные уже сопоставлены.

### 7.3. Целевые значения

Федеральная цель:

- СКР 2,10–2,15 к 2036 году;
- СКР 2,50–3,00 к 2050 году.

Для графика использовать центр коридора как целевую траекторию России, а коридор как область цели.

## 8. Обязательная логика страницы СКР

Страница СКР должна иметь режим анализа:

```text
Россия | Федеральный округ | Субъект РФ
```

### Россия

- KPI только по России.
- График: факт России, инерционный прогноз России, целевой коридор России, лаг эффекта, разрыв.
- Нет смешения с регионом.

### Федеральный округ

- На графике всегда Россия + выбранный федеральный округ.
- KPI разделены: `Россия` и `Выбранный федеральный округ`.
- Цель — федеральная рамка России.
- Для округа показывается сравнение с Россией и федеральной рамкой, но не обязательная отдельная траектория.

### Субъект РФ

- На графике всегда Россия + выбранный субъект.
- KPI разделены: `Россия` и `Выбранный субъект`.
- Цель — федеральная рамка России.
- Для субъекта показывается последний СКР, прогноз 2036/2050, разница с Россией и разница с федеральной рамкой.

### Дельты

Не использовать неоднозначную подпись `Δ к цели`.

Использовать два показателя:

```text
Отставание от цели = цель года − инерционный прогноз года
Требуемый прирост после запуска мер = цель года − прогноз на месяц начала эффекта
```

## 9. Управленческий режим интерфейса

На странице СКР обязательно сделать переключатель:

```text
Краткий вид | Подробный вид
```

По умолчанию — краткий вид.

В кратком виде показывать только главный управленческий слой. В подробном — РПН, историческую ретроспективу, таблицы, методику.

## 10. Страница «Расселение»

Название в меню: `Расселение`.

Заголовок страницы: `Расселение и ИЖС`.

Не использовать слово «дезурбанизация» как главный пользовательский термин. Использовать:

- `рост малоэтажного расселения`;
- `пригородный переход`;
- `ИЖС-сценарий`;
- `сельская и пригородная среда`.

Базовую численность брать из локального авторского прогноза. Сценарный эффект структуры расселения называть оценкой, а не полным демографическим прогнозом.

## 11. Страница «Усадьба»

Название в меню: `Усадьба`.

Требования к 3D:

- локальный Three.js;
- локальный OrbitControls;
- без внешних текстур;
- fallback SVG/PNG, если WebGL недоступен;
- дом не должен выглядеть как кубики;
- участок, дорожки, деревья, сад, основное ядро, крыша, окна, дверь, крыльцо, модули детей, модуль старшего поколения, рабочий модуль, общие пространства;
- плавное появление новых модулей;
- визуальная площадь должна соответствовать расчётной.

## 12. Страница «Маткапитал»

Название в меню: `Маткапитал`.

Runtime-файл `docs/data/matcapital_inputs.json` должен иметь `runtime_external_fetch:false`, горизонт 2050 и не содержать внешних URL. Полные ссылки источников хранятся только в `docs/data/sources.json` и документации.

Модуль показывает:

- эквивалент маткапитала 2007 года в средних зарплатах;
- жилищную модель Царьграда, привязанную к стоимости комфортного жилья;
- бюджетные обязательства как сценарную оценку сертификатов;
- цену одного потенциального рождения при заданной конверсии;
- нативные SVG-графики без внешних библиотек;
- кнопку `Скачать таблицу`.

## 13. Страница «Ипотека»

Название в меню: `Ипотека`.

Формулы:

```text
льготная часть = min(сумма кредита, лимит программы)
рыночная часть = max(0, сумма кредита − лимит программы)
платёж семьи = аннуитет(льготная часть, льготная ставка) + аннуитет(рыночная часть, рыночная ставка)
бюджетная субсидия = разница рыночной и льготной стоимости льготной части
```

Подписи:

- `Цена потенциального рождения при заданной конверсии`;
- `Номинальная субсидия за срок кредита`;
- `Скачать таблицу`.

## 14. Страница «Выплаты»

Название в меню: `Выплаты`.

Верхний итоговый вывод:

```text
При текущих настройках мера охватывает ____ матерей, даёт ____ потенциальных дополнительных рождений и требует ____ млрд ₽ до 2050 года. Цена одного потенциального рождения — ____ млн ₽.
```

Технические параметры — в блок `Дополнительные настройки`.

1. Составить план.

4. Исправить ошибки.

Запрещено завершать работу с известными failing tests, кроме случаев, когда в `RELEASE_CHECKLIST.md` явно указано и обосновано, почему проверка невозможна в среде выполнения.
